# Request for Proposal: Team Focus & Productivity Tool

**From**: Velocity Labs, Engineering Leadership
**Date**: February 2026
**Contact**: Jamie Reeves, VP of Engineering

---

## Background

Velocity Labs is a 50-person software company. Our engineering team (30
developers across 4 squads) struggles with fragmented focus time. Developers
context-switch between meetings, Slack, and deep work with no structure or
visibility into how time is actually spent.

We've tried calendar blocking and "no-meeting Wednesdays" but there's no
tool to help individuals manage their focus sessions or for leads to
understand focus patterns across the team.

## What We're Looking For

A personal focus timer — specifically a Pomodoro-style tool — that each
developer uses to structure their work sessions. Tag sessions to projects,
see where focus time goes, and build better habits.

### Key Requirements

1. **Pomodoro timer** — standard intervals (25-min work, 5-min short break,
   15-min long break after 4 sessions). Start, pause, resume, cancel.
   Audio/visual notifications on completion.
2. **Session tagging** — tag each session with a project or label (e.g.,
   "API refactor", "sprint planning prep"). Reusable tags, ability to
   change mid-session.
3. **Focus stats** — daily summaries, weekly charts, streak tracking. Help
   developers see their own patterns and stay motivated.
4. **Customization** — adjustable durations, theme preference (light/dark),
   notification toggles. Not everyone does 25/5/15.

### Nice to Have

- Export summaries (CSV or similar) for sprint retros
- Keyboard shortcuts for power users
- Minimal, distraction-free UI

## Constraints

- **Timeline**: We'd like something usable within 4-6 weeks
- **Technology**: Web-based (browser). Our team is on mixed OS (Mac, Windows,
  Linux). Modern stack preferred — React/TypeScript if possible.
- **Budget**: This is an internal tool, not a product. Small budget —
  $15K-$25K range.
- **Users**: Single-user for now (each developer runs their own). No
  backend or shared database required for v1.
- **Accessibility**: Must be keyboard-navigable and meet basic accessibility
  standards (WCAG 2.1 AA).
- **Offline**: Should work without internet. Developers sometimes work from
  planes, coffee shops, etc.

## What We Need From You

1. Confirmation you understand the scope
2. A proposed approach with technology recommendations
3. Rough timeline and cost estimate
4. How you'd break this into deliverable pieces

## Evaluation Criteria

- Simplicity of approach (don't over-engineer this)
- Understanding of developer workflow
- Clean, modern UI sensibility
- Realistic timeline
